# ปฏิบัติการสัปดาห์ที่ 3: Requirements Analysis & User Stories

## 🎯 ภาพรวมรายวิชา
- **วิชา:** ENGCE301 การออกแบบและพัฒนาซอฟต์แวร์ (Software Design and Development)
- **หัวข้อ:** การวิเคราะห์ความต้องการและการเขียน User Stories

## 📚 แนวคิดหลักที่เรียนรู้

### การวิเคราะห์ความต้องการ (Requirements Analysis)
กระบวนการในการกำหนดความคาดหวังของผู้ใช้สำหรับซอฟต์แวร์ตัวใหม่หรือที่กำลังจะปรับปรุง แบ่งออกเป็น 2 ประเภทหลัก:
1.  **Functional Requirements:** สิ่งที่ระบบต้องทำ (เช่น ระบบต้องสามารถบันทึกข้อมูลนักศึกษาได้)
2.  **Non-functional Requirements:** คุณสมบัติของระบบ (เช่น ระบบต้องประมวลผลภายใน 2 วินาที, ความปลอดภัยของข้อมูล)

### User Stories
การอธิบายคุณสมบัติย่อยๆ ของซอฟต์แวร์จากมุมมองของผู้ใช้ปลายทาง โดยมีรูปแบบมาตรฐานคือ:
> **"As a [user role], I want [goal], so that [benefit]."**

### เกณฑ์การยอมรับ (Acceptance Criteria)
เงื่อนไขที่ซอฟต์แวร์ต้องผ่านเพื่อให้ User Story นั้นถือว่าเสร็จสมบูรณ์ ช่วยให้ทีมพัฒนาและลูกค้ามีความเห็นตรงกัน

## 🛠️ เครื่องมือและเทคนิค
- **Brainstorming:** การระดมความคิดเพื่อหาความต้องการ
- **User Interviews:** การสัมภาษณ์ผู้ใช้งานจริง
- **Kanban Board:** เครื่องมือจัดการงาน (เช่น Trello, GitHub Projects) เพื่อติดตามสถานะของ User Stories

## 💻 กิจกรรมปฏิบัติการ: การสร้าง User Stories
ในสัปดาห์นี้ได้ทำการวิเคราะห์ความต้องการสำหรับ "ระบบจัดการงาน (Task Management System)" และจัดทำ User Stories พร้อมเงื่อนไขการทดสอบ
